# 阶段 0004：网关渲染

## 目标

基于根清单和所有应用清单生成共享 Caddy 网关配置，使公开应用可以通过 `<app_name>.locallab` 被路由到对应容器 upstream。

## 前置条件

- 阶段 0000 到 0003 已完成。
- `.locallab/llstk.yml` 可以读取。
- `.locallab/lab-app-<name>/llstk.yml` 可以扫描、读取和校验。
- `gitea-postgres` 模板能生成公开 upstream。

## 命令规格

```bash
llstk gateway render [--check] [--force]
llstk gateway status
```

参数：

- `--check`：只检查当前 Caddyfile 是否与渲染结果一致，不写入文件。
- `--force`：覆盖已有 Caddyfile。第一版 `gateway render` 默认可以覆盖 Caddyfile，因为它是生成文件；但如果检测到无 LocalLabStack header 的用户文件，应要求 `--force`。

## 输入来源

渲染必须读取：

- 根清单 `.locallab/llstk.yml`。
- 所有匹配 `.locallab/lab-app-*/llstk.yml` 的应用清单。

应用扫描顺序必须稳定，建议按应用名升序排序。

## 路由规则

对每个应用：

- 只渲染 `upstreams` 中 `public: true` 的项。
- 使用应用清单中的 `domain` 作为站点地址。
- 使用 upstream 的 `container` 和 `port` 作为 `reverse_proxy` 目标。
- 如果一个应用有多个公开 upstream，第一版可以报错并提示暂不支持多个公开 upstream；或者渲染多个 route。为降低复杂度，建议第一版报错。

## 输出文件

目标文件：

```text
.locallab/gateway/Caddyfile
```

生成文件必须包含 header：

```caddyfile
# Generated by LocalLabStack. Manual edits may be overwritten.
```

示例：

```caddyfile
# Generated by LocalLabStack. Manual edits may be overwritten.

{
  auto_https off
}

gitea.locallab {
  tls /certs/issued/locallab.pem /certs/issued/locallab-key.pem
  reverse_proxy gitea:3000
}

*.locallab {
  tls /certs/issued/locallab.pem /certs/issued/locallab-key.pem
  respond "Unknown LocalLabStack app" 404
}
```

## TLS 路径

第一版固定使用容器内路径：

```text
/certs/issued/locallab.pem
/certs/issued/locallab-key.pem
```

根清单中的 cert provider 和 domains 不改变 Caddyfile 中这两个路径。

## `gateway status`

至少输出：

- gateway provider。
- gateway container。
- Caddyfile 路径。
- 已发现应用数量。
- 已发现公开路由数量。
- 每条公开路由的 domain 和 upstream。

示例：

```text
provider: caddy
container: locallabstack-gateway
caddyfile: .locallab/gateway/Caddyfile
routes:
  gitea.locallab -> gitea:3000
```

## 写入策略

- Caddyfile 不存在时创建。
- Caddyfile 存在且包含 LocalLabStack header 时覆盖。
- Caddyfile 存在但不包含 header 时，默认失败并提示使用 `--force`。
- `--check` 不写文件，只比较内容。
- `--check` 一致时返回 0。
- `--check` 不一致时返回非 0，并说明需要运行 `llstk gateway render`。

## 错误场景

- 未初始化工作区时，提示先运行 `llstk init`。
- 根清单不存在或非法时失败。
- 任一应用清单非法时失败并显示路径。
- 公开 upstream 的端口非法时失败。
- 一个应用多个公开 upstream 时按本阶段决策失败。
- Caddyfile 为未标记的用户文件且未传 `--force` 时失败。

## 测试要求

至少覆盖：

- 无应用时渲染只包含 wildcard fallback。
- 创建 gitea 后渲染包含 `gitea.locallab`。
- 路由按应用名排序，输出稳定。
- `--check` 在文件一致时成功。
- `--check` 在文件不一致时失败。
- 未标记 Caddyfile 未传 `--force` 时失败。
- 非法应用清单路径出现在错误信息中。

## 验收标准

本阶段完成时，以下命令必须形成闭环：

```bash
cargo run --bin llstk -- init
cargo run --bin llstk -- app create gitea --template gitea-postgres
cargo run --bin llstk -- gateway render
cargo run --bin llstk -- gateway status
cargo run --bin llstk -- gateway render --check
```

并且 `.locallab/gateway/Caddyfile` 中存在：

```text
gitea.locallab
reverse_proxy gitea:3000
```
